Да. До 10 апреля у тебя вполне нормальный запас, чтобы закрыть базу по всем блокам программы и отдельно хорошо “пощупать” Git. В самой программе упор идёт на Linux, сеть, базы данных, Docker, Git, Ansible, расширенный Ansible и CI/CD с GitLab CI. По Git и Ansible прямо указан ожидаемый результат “умение работать с системами контроля версий и Ansible”, а по CI/CD — “понимание основ CI/CD” и “навыки работы с GitLab CI”.

Ниже дам тебе практический план с упором на то, что можно сделать руками.

## Как готовиться правильно

На каждую тему делай 3 вещи:

1. Прочитать базу.
2. Повторить команды руками.
3. Сделать маленькую практику, где ты сам что-то ломаешь и чинишь.

Идея такая: не просто “знаю команду”, а “видел, что она делает”.

## Что обязательно “пощупать” руками

### Linux

Что сделать самому:

- поставить Ubuntu в VirtualBox или WSL
- поставить nginx и управлять им через `systemctl`
- посмотреть процессы через `ps`, `top`, `htop`
- убить процесс и проверить, что стало
- создать большой файл, заполнить диск, посмотреть `df -h` и `du -sh`
- удалить файл, который держит процесс, и посмотреть, почему место не освободилось
- примонтировать что-нибудь или хотя бы посмотреть `lsblk`, `mount`, `/etc/fstab`
- написать 2–3 простых bash-скрипта

Мини-практики:

- “nginx не запущен — найди и подними”
- “диск забит — найди, чем”
- “процесс держит лог — найди через `lsof`”

### Сеть

Что сделать самому:

- посмотреть `ip a`, `ip route`
- попинговать localhost, шлюз, внешний IP, домен
- сравнить `ping 8.8.8.8` и `ping google.com`
- сделать `nslookup` или `dig`
- сделать `traceroute`
- посмотреть ARP-таблицу
- в Windows посмотреть `route print`, в Linux `ip route`

Мини-практики:

- понять разницу между “нет DNS” и “нет сети”
- понять, когда появляется `No route to host`
- найти свой шлюз и объяснить, зачем он нужен

### PostgreSQL

Что сделать самому:

- установить PostgreSQL
- подключиться через `psql`
- создать БД
- создать пользователя
- дать права
- создать таблицу
- вставить пару записей и сделать `SELECT`

Мини-практика:

- поднять PostgreSQL и подключиться к нему локально
- потом попробовать подключение по сети, если сможешь

### Docker

Что сделать самому:

- установить Docker
- запустить `nginx` в контейнере
- запустить `postgres` в контейнере
- посмотреть контейнеры, логи, образы
- удалить контейнер
- пробросить порт
- сделать свой простой Dockerfile

Мини-практики:

- собрать контейнер с простым приложением
- запустить контейнер и проверить, что он доступен через браузер или `curl`

### Ansible

Что сделать самому:

- поставить Ansible
- сделать inventory из localhost или двух VM
- проверить `ansible all -m ping`
- написать playbook, который ставит nginx
- playbook, который копирует конфиг
- playbook, который перезапускает сервис

Мини-практика:

- поднять nginx вручную
- потом автоматизировать это через Ansible

### CI/CD

Что сделать самому:

- создать простой `.gitlab-ci.yml`
- сделать стадии `test`, `build`, `deploy`
- хотя бы локально понять логику
- если GitLab нет, просто написать YAML и понять, что делает каждая job

Мини-практика:

- пайплайн, который проверяет Python-файл или shell-скрипт
- пайплайн, который собирает Docker image

# Отдельно: что именно “щупать” по Git

Вот тут лучше всего не читать, а прям жить в Git несколько дней.

## База, которую надо уметь руками

Нужно спокойно пользоваться:

- `git init`
- `git clone`
- `git status`
- `git add`
- `git commit`
- `git log`
- `git diff`
- `git branch`
- `git checkout`
- `git switch`
- `git merge`
- `git pull`
- `git push`
- `git reset`
- `git restore`

## Практика по Git, которую советую сделать обязательно

### 1. Создай свой тестовый репозиторий

Сделай папку с простым проектом:

- `README.md`
- `app.py` или `script.sh`
- папку `configs`
- `.gitignore`

Что потрогаешь:

- инициализация репозитория
- первый коммит
- история коммитов

### 2. Научись делать маленькие осмысленные коммиты

Не один большой коммит “всё сразу”, а отдельно:

- добавил README
- добавил скрипт
- исправил баг
- добавил конфиг

Это важно, потому что на стажировке любят смотреть, понимаешь ли ты логику работы с историей.

### 3. Потренируй ветки

Сделай так:

- `main`
- `feature/add-logging`
- `feature/fix-readme`

Что сделать:

- создать ветку
- внести изменения
- сделать коммит
- смержить в `main`

### 4. Специально устрой merge conflict

Это очень полезно.

Как сделать:

- в `main` измени одну строку в файле
- в другой ветке измени ту же строку
- попробуй сделать merge

Потом:

- открой конфликт
- руками исправь
- закоммить результат

Если ты один раз руками решишь конфликт, Git станет намного понятнее.

### 5. Потренируй `git diff`

Сравни:

- рабочую директорию и индекс
- индекс и последний коммит
- две ветки
- два коммита

Тебе важно начать видеть: “что именно я поменял”.

### 6. Потренируй откаты

Сделай отдельный тестовый репозиторий и попробуй:

- изменить файл и отменить изменения
- удалить файл и вернуть его
- откатить staged changes
- откатить последний коммит в учебных целях

Важно не заучить команды, а понять разницу:

- файл ещё не добавлен
- файл уже в staging
- коммит уже сделан

### 7. Потренируй работу с удалённым репозиторием

Если есть GitHub/GitLab:

- создай репозиторий
- подключи `origin`
- сделай push
- клонируй в другую папку
- внеси изменения
- pull/push обратно

### 8. Сделай учебный workflow “как на работе”

Вот хороший мини-сценарий:

- есть `main`
- создаёшь ветку `feature/nginx-config`
- меняешь файл
- коммитишь
- пушишь ветку
- делаешь merge request мысленно или реально
- мержишь в `main`

### 9. Посмотри `.gitignore`

Сделай так, чтобы Git не отслеживал:

- `__pycache__/`
- `.env`
- `*.log`

Это часто спрашивают косвенно.

### 10. Посмотри историю

Научись читать:

- кто что поменял
- в каком коммите
- в каком файле

Это даст уверенность.

# Очень полезные мини-проекты для Git

## Вариант 1. Репозиторий с bash-скриптами

Сделай репозиторий `devops-practice` и добавляй туда:

- `check_nginx.sh`
- `cleanup_logs.sh`
- `disk_check.sh`

В процессе потрогаешь:

- ветки
- коммиты
- pull/push
- конфликты
- `.gitignore`

## Вариант 2. Репозиторий с Docker + Git

Сделай:

- `Dockerfile`
- `app.py`
- `requirements.txt`
- `.gitlab-ci.yml`

Тогда одной практикой ты закроешь сразу Git, Docker и CI/CD.
